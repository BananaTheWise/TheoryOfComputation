Like Math But Not Math

Project Architecture and Module Summary

1. Project Overview

The Like Math But Not Math is a comprehensive Python-based desktop application built using PyQt5. It serves as an educational and analytical tool for the Theory of Computation. The application seamlessly translates between different representations of formal (regular) languages. Users can input a Regular Expression, a Deterministic Finite Automaton (DFA), a Non-deterministic Finite Automaton (NFA), a Context-Free Grammar (CFG), or a finite set of strings. The application then normalizes this input into a minimal DFA, visually renders the automaton graph, and simultaneously derives all other representations, including an English-language description of the recognized pattern.

2. Detailed File Breakdown

Root Directory (Orchestration & Setup)

main.py

Role: Application Entry Point.

Description: Bootstraps the PyQt5 application. It initializes the global application settings, applies a custom dark theme styling using QPalette and stylesheets, displays a custom splash screen with the project's branding, and finally instantiates and launches the MainWindow.

Technical Details: The script sets the execution path to correctly resolve internal modules. It uses QPainter to programmatically generate a splash screen image in-memory. The set_dark_theme() method overrides the OS default styling by applying a custom QPalette alongside a robust embedded CSS string, mapping variables to PyQt widgets like buttons, scrollbars, and tabs for a uniform "Fusion" dark aesthetic. Finally, it delegates execution to the main Qt event loop.

controller.py

Role: The Backend API / Logic Bridge.

Description: Acts as the intermediary between the GUI and the underlying computational algorithms. It defines custom parsing functions (like parse_dfa_from_text and parse_cfg_from_text) to interpret the raw user input from the UI. The core convert() function takes the parsed input, delegates it to the appropriate algorithms to generate a minimal DFA, and then back-computes all equivalent language representations (Regex, CFG, generated strings, English description) returning a unified dictionary to the UI.

Technical Details: Acts as a Facade pattern over the complex Core modules. It handles the parsing phase by splitting strings and applying semantic rules (e.g., verifying transition structures state,symbol -> next). The convert() function centralizes exception handling, catching ValueErrors or RecursionErrors and piping them back as a safe 'error' string instead of crashing the UI. It also includes utility functions like strings_to_dfa() which builds a Trie out of a finite set of strings and translates the Trie nodes into DFA states.

requirements.txt & README.md

Role: Documentation and Dependencies.

Description: requirements.txt lists the external dependencies (PyQt5, pytest, graphviz). README.md provides user instructions on installation, execution, and syntax rules for inputting various formats.

Technical Details: These standard configuration files define the runtime environment. The README strictly defines the DSL (Domain Specific Language) expected by the UI text boxes, ensuring users understand how to represent mathematical concepts (like using ε, epsilon, or eps for the empty string).

conftest.py

Role: Test Configuration.

Description: Sets up the Python system path prior to test execution, ensuring that pytest can successfully locate and import modules from the project's root directory.

Technical Details: This ensures that standard testing frameworks do not fail due to relative import ambiguities, seamlessly modifying sys.path at the root level before any test suite loads.

Core Module (/Core) - Theoretical Algorithms

automata.py

Role: Fundamental Data Structures.

Description: Contains the DFA and NFA Python dataclasses. This file provides intrinsic automaton functionality such as simulating a string against the machine (accepts()), epsilon-closures for NFAs, subset construction internally for NFA to DFA conversion, DFA state reachability parsing, and methods to export the automatons to Graphviz DOT formats for rendering.

Technical Details: Built heavily on Python sets and dictionaries for O(1) average lookup times. The transition tables are represented as Dict[Tuple[str, str], str] (or Set[str] for NFAs). It includes an integrated Hopcroft minimization algorithm in the DFA class, partitioning states into accepting and non-accepting, and iteratively splitting them based on transition equivalence. Epsilon-closures use stack-based DFS traversal. Both classes provide native serialization via to_dict()/from_dict() for JSON portability and to_graphviz() for visual exports.

regex_parser.py

Role: Lexer and Abstract Syntax Tree (AST) Generator.

Description: Implements a custom recursive descent parser that breaks down a regular expression string into an Abstract Syntax Tree. It handles operator precedence (Star/Plus/Question Mark > Concatenation > Union) and identifies syntax errors.

Technical Details: Implements a classic formal grammar parsing algorithm without relying on Python's native re library. It consumes characters linearly, constructing polymorphic AST nodes (LiteralNode, StarNode, UnionNode, etc.). The recursive structure inherently enforces the order of operations, where grouping via parentheses recursively calls the parse_union() root. It explicitly guards against malformed syntax like dangling operators, unmatched parentheses, or consecutive quantifiers.

thompson.py

Role: Thompson's Construction Algorithm.

Description: Takes the AST produced by regex_parser.py and traverses it to construct an equivalent NFA. It contains the logic for joining sub-NFAs together using epsilon transitions according to Thompson's theoretical rules.

Technical Details: Recursively maps each AST node to a small NFAFragment. It uses inductive proofs from automata theory: combining fragments by wiring the accept state of one fragment to the start state of another using ε-transitions. A StateCounter maintains globally unique identifiers (q0, q1, etc.) throughout the recursive calls. The algorithm natively maps extended operators like + (one or more) and ? (zero or one) into standard NFA logic.

subset.py

Role: Subset Construction & Minimization.

Description: Houses two critical algorithms. First, it implements formal Subset Construction (Powerset construction) to determinize an NFA into a DFA. Second, it implements Hopcroft's partition refinement algorithm (dfa_minimize) to reduce the DFA to its most optimal, minimal number of states.

Technical Details: Powerset Construction computes the ε-closure of a starting set of states and creates a new aggregate state. By treating frozensets of NFA states as singular DFA states, it iteratively evaluates all possible inputs to discover newly reachable sets until closure. The Hopcroft minimization algorithm is implemented optimally by maintaining a working set of partitions W and evaluating discriminators to split equivalent states down to their foundational mathematical equivalence classes, stripping away any redundancy.

state_elim.py

Role: State Elimination Algorithm.

Description: Implements the reverse conversion from a DFA back into a Regular Expression. It constructs a Generalized NFA (GNFA), systematically rips out states, and builds up a complex regular expression along the remaining transition edges.

Technical Details: Transforms the standard DFA into a GNFA by defining a singular new start state and a singular new accept state connected via ε-transitions. It selectively eliminates states one-by-one. When state q_mid is ripped, the transition connecting q_in to q_out is rewritten as a regex: R_in_mid (R_mid_mid)* R_mid_out, unioned with the existing direct transition. The file includes heavy string manipulation utilities to inject parentheses dynamically only when necessary to preserve precedence during concatenations.

grammer.py

Role: Context-Free Grammar Conversions.

Description: Contains functions to translate a regular language back and forth between a DFA and a Context-Free Grammar. It establishes variable/terminal mappings and production rules (dfa_to_cfg) and reverses the process by turning production rules into NFA states (cfg_to_nfa).

Technical Details: Translates state machines into Right-Linear Grammars (a subset of CFG strictly equivalent to regular languages). For DFA to CFG, every state becomes a Non-Terminal Variable (e.g., V_q0), and transitions become production rules (V_q0 → a V_q1). Accepting states receive an ε production. Conversely, CFG to NFA parses production strings, assigning the left-hand variable as the origin state and mapping the right-hand side to a transition and destination state, constructing a dummy final state for terminal-only rules.

enumerator.py

Role: String Generation & Counting.

Description: Analyzes a DFA to lazily generate the actual strings that belong to the language. It uses a Breadth-First Search (BFS) to prevent infinite loops, handles dead/trap states gracefully, and limits output based on string length. It also includes dynamic programming logic for exact combinations counting.

Technical Details: Employs a generator pattern (yield) to support lazy evaluation, crucial for infinite languages. It optimizes the BFS by pre-computing reachability; if an accept state cannot be reached from the current branch, the branch is aggressively pruned to prevent memory explosions traversing deep into 'trap' structures. Additionally, the count_strings() uses dynamic programming, building a 2D matrix of lengths and states to calculate exact combinations mathematically without actually simulating them.

GUI Module (/GUI) - User Interface

main_window.py

Role: Primary User Interface.

Description: Assembles the PyQT5 widgets, managing the Left Panel (dynamic inputs for Regex, structured tables for FA/CFG), the Center Panel (Canvas rendering), and the Right Panel (tabbed outputs for Regex, CFG, Strings, English). It utilizes a QThread worker to keep the UI responsive while expensive DFA algorithms run in the background.

Technical Details: Manages state using PyQT signals and slots. Input modes are toggled via a QStackedWidget-like behavior, swapping out plaintext inputs for QTableWidgets. Because converting large automata or enumerating strings can block the main thread, the actual convert() execution is wrapped in a ConvertWorker class extending QThread. The results are emitted back via a custom pyqtSignal to safely update the GUI asynchronously.

dfa_canvas.py

Role: Graph Visualizer.

Description: A custom widget that takes a DFA object and uses the graphviz library to render it into an image. It handles zooming, panning, and provides the groundwork for step-by-step visual simulation of string testing (highlighting the current active state in yellow/gold).

Technical Details: Leverages the Graphviz dot engine to compute optimal node layouts dynamically. The dot file is written to a temporary directory and compiled to both PNG (for display via QPixmap) and plain text (to parse node coordinates). Interactive zooming is handled by applying scale transformations to the base Pixmap. During interactive string simulation, the DFA graph is repeatedly re-rendered with modified node styling properties (altering fillcolors to highlight active states).

Nlg Module (/Nlg) - Natural Language Generation

describer.py

Role: Human-Readable Language Analysis.

Description: Programmatically inspects the structure and accepted strings of a DFA to deduce its purpose in plain English. Uses mathematical heuristics to detect if the language accepts everything, nothing, has a fixed length, demands a specific prefix/suffix, or counts specific symbols (Modulo arithmetic). This bridges the gap between raw math and human comprehension.

Technical Details: Implements pattern recognition on the DFA structure. Emptiness is checked by verifying reachability from the start state to any accept state. Finiteness is verified via cycle detection (DFS with recursion stack tracking). Fixed-length checks evaluate standard string enumerations. Modulo behavior is detected by iterating states and checking if symbols form a perfect, continuous cycle covering all states with a single accepting modulo remainder. It strings together these mathematical deductions to dynamically template English sentences.

Tests Module (/tests) - Quality Assurance

A comprehensive suite of Pytest files designed to test the application's mathematical algorithms to ensure complete theoretical accuracy:

  • test_automata.py: Tests DFA/NFA object behaviors, state transitions, epsilon closures, and JSON serialization.
  • test_regex_parser.py: Verifies AST generation, correct operator precedence, and syntax error catching.
  • test_thompson.py: Asserts that Regex successfully compiles to correct and operational NFAs.
  • test_subset.py: Validates NFA to DFA determinization and state minimization algorithms.
  • test_state_elim.py: Verifies DFA to Regex backward compilation via round-trip language testing.
  • test_grammer.py: Confirms production rules are accurately mapped from DFA to CFG and vice versa.
  • test_enumerator.py: Checks accurate lexicographical string output bounds and counting logic.
  • test_describer.py: Verifies the Nlg module accurately deduces plain-English heuristics from DFAs.
  • test_integration.py: Tests the controller.py conversions from end-to-end to simulate full user flows.
